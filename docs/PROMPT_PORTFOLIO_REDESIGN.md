# Portfolio Redesign Prompt (Gemini CLI)

## 목표

칙칙하고 상막한 다크 포트폴리오를  
**밝고 정돈된 Technical PM 포트폴리오**로 리디자인한다.

- 화이트 기반이지만 가볍지 않게
- 구조가 먼저 보이는 레이아웃
- “일 맡겨도 될 PM” 인상

---

## 공통 역할 설정 (최초 1회)

너는 **시니어 프론트엔드 디자이너 + 엔지니어**다.

### 디자인 방향

- 완전 화이트 ❌ → 소프트 화이트 / 라이트 그레이 베이스
- 카드 기반 레이아웃 유지
- 포인트 컬러는 **테크 블루 1개만** 사용
- 읽기 전에 구조가 보이도록 설계

### 원칙

- 전역 디자인 토큰(CSS 변수) 먼저 정의
- 섹션 간 여백 확대
- 문단은 최대 3줄, 나머지는 리스트 처리
- 질문하지 말고 합리적 가정으로 진행

### 결과물

1. 디자인 변경 요약
2. 수정된 코드 (전체 또는 diff)
3. 적용 방법

---

## STEP 1. 밝은 테마 디자인 토큰 정의

### 요구사항

#### 배경

- 전체 배경: `#f8fafc`
- 섹션 배경: `#ffffff`

#### 카드

- 카드 배경: `#ffffff`
- 카드 테두리: `#e5e7eb`
- Hover 시 부드러운 shadow

#### 텍스트

- 메인 텍스트: `#111827`
- 보조 텍스트: `#4b5563`
- 약한 텍스트: `#6b7280`
- 섹션 타이틀은 명확한 대비

#### 포인트 컬러

- Primary: `#2563eb`
- 링크 / 버튼 / 강조 요소에만 사용

#### 버튼

- Primary / Secondary / Ghost 상태 정의
- Hover 시 미세한 색 변화

#### 공통 스타일

- Border-radius: `12px`
- Shadow는 매우 소프트하게

### 결과

- `:root` 기준 CSS 변수 전체 코드 제공

---

## STEP 2. Hero 섹션 리디자인

### 목표

- 이력서 첫 장 같은 첫인상
- Technical PM 정체성이 즉시 보이게

### 구조

- 이름: 가장 크게
- 직무: 서브 타이틀
- 슬로건: 한 줄
- 설명 문단: 최대 2줄
- GitHub / 프로젝트 버튼 강조

### 텍스트 예시

김현중
Technical PM · 구조 설계형 서비스 기획자
아이디어를 실행 가능한 구조로 설계합니다.

- 과한 애니메이션 금지
- 미세한 fade-in 허용

---

## STEP 3. Core Skills 섹션 개선

### 변경 방향

- 무거운 카드 ❌
- 밝은 배지(Chip) 스타일 ⭕

### 요구사항

- 한 줄 요약형 스킬
- 아이콘 또는 컬러 포인트
- Hover 시 배경 강조
- Hover/클릭 시 한 줄 설명 노출

### 스킬 예시

- 요구사항 구조화
- API · DB 흐름 설계
- 데이터 기반 기획
- 운영 관점 문제 정의

---

## STEP 4. Projects 섹션 리디자인

### 요구사항

- 카드 간 여백 확대
- 프로젝트명 + 한 줄 요약 강조
- 문제 / 역할 / 해결 / 결과는 리스트 구성
- 결과는 숫자/성과 중심
- GitHub 버튼은 Primary 스타일

### 원칙

- 정보 밀도 유지
- 시각적 부담 최소화

---

## STEP 5. 전체 톤 점검

### 체크리스트

- 칙칙한 색상 잔존 여부
- 각 섹션의 시각적 시작점 존재
- 텍스트 대비 WCAG 기준 충족
- PM 포트폴리오로서 신뢰감 유지

### 최종 기준

> “밝지만 가벼워 보이지 않는  
> Technical PM 포트폴리오”

---

## 완료 후 기대 인상

- ❌ 학생 포트폴리오
- ❌ 디자인 중심 포폴
- ✅ 구조를 이해하는 PM
- ✅ 개발자와 바로 협업 가능한 기획자
